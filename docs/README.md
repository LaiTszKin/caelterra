# 專案文檔

## 文檔結構

| 目錄 | 用途 | 維護原則 |
|------|------|---------|
| `docs/features/` | 功能使用說明 — BDD 格式描述使用者面向行為 | 程式碼變更影響功能行為時更新 |
| `docs/architecture/` | 架構設計文件 — 模組邊界、資料流程、設計決策 | 新增/重構模組時更新 |
| `docs/principles/` | 開發慣例 — 命名規則、技能開發、測試慣例 | 團隊共識改變時更新 |
| `docs/plans/` | 活動中 spec（尚未完成/尚未歸檔） | spec 完成後移入 docs/archive/ |
| `docs/archive/` | 已歸檔 spec — 供日後查閱 | 不再修改 |

## 維護指引

- **證據追溯**：文檔中每條記述必須附上來源檔案與行號區間。無法從程式碼直接證明的 claim 標記為 `[INFERRED]`。
- **LLM 安全原則**：產生文檔時，只餵給 LLM 結構化中繼資料（檔案列表、模組邊界、API 端點、函式簽名），不傳輸原始碼完整內容。
- **增量更新**：當程式碼變更時，只重新產生受影響的文檔區段。可使用 `git diff` 識別變更範圍，再決定哪些 `.md` 檔案需要更新。
- **Drift detection**：建議每月比對文檔與實際程式碼，確認無重大偏離。發現 drift 時只修補受影響章節，不全面重寫。
