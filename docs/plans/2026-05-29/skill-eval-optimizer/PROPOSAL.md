# Proposal: 自動化技能評測與優化系統 (Skill Eval & Optimizer)

- **Date**: 2026-05-29
- **Source**: 由 `discuss` 技能通過結構化對話釐清需求後產出

---

## 1. 功能範圍

### 要做的事

- 針對 `spec` 技能，建立一套自動化評測流程：出題 → 執行 → 評分 → 優化
- 維護一個核心人工題庫（覆蓋關鍵場景），搭配 LLM 動態生成題目變體增加覆蓋面
- 從三個維度評分：指令遵循度、工具調用質量、結果質量（由大模型擔任評分模型）
- 評測時記錄模型的思考軌跡、工具調用軌跡、最終輸出，作為評分模型的輸入
- 優化器根據評分結果和問題發現，自動修改技能目錄下的 `SKILL.md`、`scripts/`、`references/`、`assets/` 文件
- 支援兩種模式：快速模式（3-5 題，開發迭代用）和標準模式（8-12 題，PR 閘門用）
- 支援 CLI 手動觸發（如 `apltk eval spec`）和 CI/CD PR 閘門自動觸發
- 輸出評測報告（Markdown）+ 優化 diff，讓開發者先理解問題再看修改建議
- 評分模型可由使用者透過 `.env` 配置
- 讀取操作（讀檔、搜尋等）真實執行；寫入操作（寫檔、API 呼叫等）使用 mock 替代

### 明確不做的事

- 不一次做全部技能 — 先聚焦 `spec`，驗證概念後再考慮擴展
- 不做全自動閉環迭代優化（改寫 → 重測 → 再改寫直到分數達標），避免迭代次數和成本不可控
- 不做排程定時評測 — 只支援手動觸發和 PR 觸發
- 不做深度模式（20+ 題）— 目前需求不明確
- 不做跨專案的通用評測框架 — 先放在 Apollo Toolkit 內部
- 不在評測系統初期就依賴量化分數作為唯一成功指標

## 2. 使用者場景

### 目標使用者

Apollo Toolkit 的唯一維護者（目前只有你自己），在維護和改進技能時使用。

### 典型流程

**場景 A：開發時快速迭代**
1. 開發者修改了 `spec` 技能的 SKILL.md 或相關文件
2. 在終端機執行 `apltk eval spec`
3. 系統從題庫抽取 3-5 道題目，必要時由 LLM 產生變體
4. 模型逐題執行 spec 技能，記錄思考軌跡、工具調用、最終輸出
5. 評分模型對每題從三個維度打分，產出結構化評測報告
6. 開發者閱讀報告，了解問題所在
7. 開發者可選擇讓優化器基於評測結果自動產生優化 diff
8. 開發者審查 diff 後決定是否採納

**場景 B：PR 閘門**
1. 開發者提交修改技能檔案的 PR
2. CI 自動執行標準模式評測（8-12 題）
3. 若分數低於門檻或出現嚴重回歸，CI 回報失敗並附上評測報告
4. 開發者根據報告修復問題後重新推送

### 成功標準

- 開發者修改技能後，能在一鍵之內獲得評測結果和改善方向
- 優化器建議的 diff 能實際改善技能表現（初期以主觀判斷為主）
- PR 閘門能有效攔截品質回歸

### 失敗處理

- 若評分模型 API 不可用：顯示明確的連線錯誤訊息，不阻塞開發流程
- 若題庫為空：提示使用者需先建立題庫
- 若優化器產出語法錯誤的 diff：保留原始文件備份，diff 需經人工審查後才生效

## 3. 約束條件

- **評分模型配置**：使用者透過 `.env` 配置評分模型（API key、模型名稱等），確保靈活性
- **上下文隔離**：每次評分呼叫必須在獨立上下文中進行，避免被評測的模型上下文污染評分模型的判斷
- **工具模擬策略**：讀取操作（Read、Bash-readonly、Grep 等）真實執行，確保評分能反映模型定位資訊的能力；寫入操作（Write、Edit、Bash-write 等）mock 掉，避免副作用
- **系統位置**：作為 Apollo Toolkit 的內建工具或技能，放在專案內部
- **成本可控**：快速模式 3-5 題、標準模式 8-12 題，避免 API 呼叫成本失控

## 4. 業務價值

### 解決的問題

技能（SKILL.md）的品質改進目前完全依賴人工判斷和手動調整，缺乏系統性的評測反饋和數據驅動的優化建議。

### 為什麼現有方案不夠

- 修改技能後只能靠「感覺」判斷是否有改善，無法系統性驗證
- 沒有自動化回歸測試，技能可能在某場景變好但在其他場景變差
- 技能優化依賴開發者的經驗和直覺，缺乏結構化的問題診斷

### 成功指標

- **初期（定性）**：優化後的 `spec` 技能在實際使用中的產出品質有肉眼可見的提升
- **後期（定量）**：累積足夠評測數據後，定義可量化的品質門檻（例如評測總分不低於 X 分）

---

## 5. 剩餘未解決的問題

- 人工核心題庫的初始題目數量與具體內容尚未定義（建議在 spec 階段與技能開發者一起設計）
- 評測報告的具體格式和欄位（在 design 階段細化）
- PR 閘門的具體分數門檻（待系統跑通、累積基準線數據後再定義）
- `.env` 中評分模型的具體配置欄位設計（在 design 階段定義）
