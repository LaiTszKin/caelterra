# Worker Prompt: T2.1-cli-parser-help

- **Source task**: T2.1 Auto-update CLI parser, types, and help text

---

## 1. Mission & Rules

### Mission

Add a public `auto-update` CLI command surface with `enable`, `disable`, `status`, and `run` actions, including parser types and help text.

### Rules

- Follow the Scope in Section 5 — only modify files listed as Allowed.
- Do not wire command execution in `packages/cli/index.ts`; that is a later task.
- Do not add new dependencies.
- Workers are leaf nodes — do not spawn sub-workers.

---

## 2. Context

### Input Files

- `docs/plans/2026-06-16/background-auto-update/SPEC.md` — CLI status/enable/disable requirements.
- `packages/cli/types.ts` — `ParsedArguments` command union currently supports install, uninstall, tool, tools-help.
- `packages/cli/parsers/types.ts` — parser-specific command unions.
- `packages/cli/parsers/install-parser.ts` and `uninstall-parser.ts` — existing parser style.
- `packages/cli/help-text-builder.ts` — public help text builder.
- `test/cli/install-args-parser.test.js`, `test/cli/uninstall-args-parser.test.js`, `test/cli/dispatch-table.test.js`, `test/cli/help-text-builder.test.js` — existing test style.

### Background

The user must be able to run commands equivalent to:
- `apltk auto-update status`
- `apltk auto-update enable`
- `apltk auto-update disable`
- `apltk auto-update run`
- `apltk auto-update --help`

The parser should follow the existing command parser pattern and integrate cleanly into `ParsedArguments`, but actual execution belongs to another worker.

## 3. Tasks

### `packages/cli/types.ts` — extend parsed argument model

1. Open `packages/cli/types.ts`.
2. Extend `ParsedArguments.command` to include `'auto-update'`.
3. Add `autoUpdateAction: 'enable' | 'disable' | 'status' | 'run' | null`.
4. Extend `helpTopic` to include `'auto-update'`.

### `packages/cli/parsers/types.ts` — add parser command type

1. Open `packages/cli/parsers/types.ts`.
2. Add `AutoUpdateAction` type.
3. Add `AutoUpdateCommand` interface with command `'auto-update'`, action, `showHelp`, `toolkitHome`, and help topic.
4. Add `AutoUpdateCommand` to `ParsedCommand`.

### `packages/cli/parsers/auto-update-parser.ts` — create parser

1. Add a new parser file.
2. Support command forms:
   - `auto-update status`
   - `auto-update enable`
   - `auto-update disable`
   - `auto-update run`
   - `auto-update --home <path> status`
   - `auto-update status --home <path>`
   - `auto-update --help`
3. Default action should be `status` when no action is supplied.
4. Reuse existing parser utility patterns for `--home` and unknown flags.
5. Implement `toParsedArguments` with all fields required by `ParsedArguments`.

### `packages/cli/help-text-builder.ts` — add auto-update help

1. Add `autoUpdate()` method to `HelpTextBuilder`.
2. Include usage, behavior notes, options, and examples.
3. Mention:
   - Background auto-update is default-on after install.
   - `disable` only disables scheduled background updates; manual update remains available.
   - `run` executes one update attempt without updating the CLI package.

### Tests

1. Add `test/cli/auto-update-args-parser.test.js` covering valid/invalid parser cases.
2. Update `test/cli/help-text-builder.test.js` or add a focused test to assert auto-update help includes enable/disable/status/run and default-on wording.
3. Do not modify CLI dispatch tests yet unless they only assert types; execution dispatch is owned by T3.1.
