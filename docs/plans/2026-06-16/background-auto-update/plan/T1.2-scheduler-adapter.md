# Worker Prompt: T1.2-scheduler-adapter

- **Source task**: T1.2 Cross-platform scheduled task adapter

---

## 1. Mission & Rules

### Mission

Implement a cross-platform scheduler adapter that can generate and execute enable, disable, and status operations for macOS, Linux, and Windows without mutating real OS schedulers in tests.

### Rules

- Follow the Scope in Section 5 — only modify files listed as Allowed.
- If a file or function cannot be found, or the implementation is blocked, report it immediately.
- Do not add new dependencies.
- Workers are leaf nodes — do not spawn sub-workers.

## 3. Tasks

### `packages/cli/auto-update-scheduler.ts` — create scheduler module

1. Add a new file `packages/cli/auto-update-scheduler.ts`.
2. Define exported types:
   - `SchedulerPlatform = 'darwin' | 'linux' | 'win32'`.
   - `SchedulerActionResult` with `registered: boolean`, `platform: string`, `message?: string`.
   - `SchedulerCommandExecutor = (command: string, args: string[], options?: { env?: NodeJS.ProcessEnv }) => Promise<{ stdout: string; stderr: string }>`
   - `SchedulerOptions` with `toolkitHome`, `runnerCommand`, `env`, and optional `exec`.
3. Add helpers:
   - `buildRunnerCommand({ nodePath, cliPath, toolkitHome }): string[]` or equivalent typed structure for invoking `apltk auto-update run --home <toolkitHome>`.
   - `registerAutoUpdateTask(options): Promise<SchedulerActionResult>`.
   - `unregisterAutoUpdateTask(options): Promise<SchedulerActionResult>`.
   - `getAutoUpdateTaskStatus(options): Promise<SchedulerActionResult>`.
4. Implement platform-specific behavior:
   - macOS: write a launchd plist under the user's `~/Library/LaunchAgents` and call `launchctl bootstrap`/`bootout` or equivalent user-domain commands. Include stable label and stdout/stderr paths from `auto-update-state` if available.
   - Linux: write user systemd `.service` and `.timer` unit files, then call `systemctl --user daemon-reload`, `enable --now`, `disable --now`, and status/list checks.
   - Windows: call `schtasks /Create`, `/Delete`, and `/Query` with a stable task name and daily schedule.
5. Surface command failures by throwing an Error that includes the platform and action.
6. Keep all command execution behind the injected executor for tests.

### `test/cli/auto-update-scheduler.test.js` — add tests

1. Add a new test file `test/cli/auto-update-scheduler.test.js`.
2. Use fake temp homes and a fake executor that records commands.
3. Cover:
   - macOS registration command shape and plist path.
   - Linux registration command sequence and unit file paths.
   - Windows registration command contains `/SC DAILY` and stable task name.
   - Disable operations are idempotent enough to handle missing tasks as non-corrupting failures only when platform command indicates success.
   - Command failure throws and includes platform/action.
