### Background

The existing `cli.js` (1214 lines) has individual verb functions for each entity type:
- `verbFeature(action, flags, projectRoot, io)` — at line 349
- `verbSubmodule(action, flags, projectRoot, io)` — at line 369
- `verbEdge(action, flags, projectRoot, io)` — at line 519

The `add` and `remove` commands should wrap these existing functions.

**Relationship flag to existing parameter mapping:**
- `add feature <name> --depends-on <target>` → `verbFeature('add', { slug: <name>, 'depends-on': <target> }, ...)`
- `add module <name> --part-of <feature>` → `verbSubmodule('add', { feature: <feature>, slug: <name> }, ...)`
- `add module <name> --part-of <feature> --depends-on <target>` → `verbSubmodule('add', ...)` + `verbEdge('add', { from: '<feature>/<name>', to: <target> }, ...)`
- `add relation <name> --data-flow-to <target>` → `verbEdge('add', { from: <name>, to: <target>, kind: 'data-row' }, ...)`
- `add relation <name> --implements <target>` → `verbEdge('add', { from: <name>, to: <target>, kind: 'call' }, ...)`
- `add relation <name> --deployed-on <target>` → `verbEdge('add', { from: <name>, to: <target>, kind: 'call' }, ...)`
- `remove feature <name>` → `verbFeature('remove', { slug: <name> }, ...)`
- `remove module <name> --part-of <feature>` → `verbSubmodule('remove', { feature: <feature>, slug: <name> }, ...)`
- `remove relation <from> <to>` → `verbEdge('remove', { from: <from>, to: <to> }, ...)`

**Batch mode**: When multiple entity blocks are provided, apply each with `--no-render` injected into flags, then call `cli.runRender()` once at the end.

The dispatch switch is at lines 1161-1181 with `switch (verb)`.

## 3. Tasks

### `skills/init-project-html/lib/atlas/cli.js` — Add `verbAdd` function

1. Open `skills/init-project-html/lib/atlas/cli.js`
2. Locate the section after the existing verb functions (after `verbActor` at line 596, before `verbValidate` at line 613)
3. Add `verbAdd(args, flags, projectRoot, io)` function:

   The function should:
   - Read entity type from positional args[0]
   - Read entity name from positional args[1]
   - Based on entity type (`feature`, `module`, `relation`), construct the appropriate flags
   - For `feature`: call `verbFeature('add', { slug: <name>, ...relationFlags }, projectRoot, io)`
   - For `module`: call `verbSubmodule('add', { feature: <feature>, slug: <name>, ... }, projectRoot, io)`. If `--depends-on` provided, also call `verbEdge('add', ...)` for the dependency.
   - For `relation`: call `verbEdge('add', { from: <name>, to: <target>, kind: <mapped>, ... }, projectRoot, io)`. Infer endpoint kind from flags.
   - If entity type is unknown, throw `new Error('...')` listing supported types.
   - Relation flag mapping:
     - `--depends-on <target>` → for features, use `verbFeature`'s `--depends-on`; for modules, create an edge `from: '<feature>/<name>' to: <target>`
     - `--data-flow-to <target>` → `kind: 'data-row'`
     - `--implements <target>` → `kind: 'call'`
     - `--deployed-on <target>` → `kind: 'call'`
   - Pass through `--spec`, `--no-render`, `--project`, `--dry-run`, `--evidence` flags

4. **Batch mode support**: After the entity type is consumed, check if there are additional positional args. If so, process each pair as a separate entity. Suppress render on intermediate calls by injecting `--no-render`, then call render once at the end. The batch format is:
   ```
   apltk architecture add feature payment --depends-on order module api --part-of payment
   ```
   Where each entity is defined by: `<entity-type> <name> [flags...]` in a flat sequence.

### `skills/init-project-html/lib/atlas/cli.js` — Add `verbRemove` function

1. Add `verbRemove(args, flags, projectRoot, io)` function:

   The function should:
   - Read entity type from positional args[0]
   - Read entity name from positional args[1]
   - For `feature`: call `verbFeature('remove', { slug: <name> }, projectRoot, io)`
   - For `module`: call `verbSubmodule('remove', { feature: <--part-of>, slug: <name> }, projectRoot, io)`. Require `--part-of` flag.
   - For `relation`: call `verbEdge('remove', { from: <from>, to: <to> }, projectRoot, io)`. Require `--from` and `--to` flags.
   - If entity type is unknown, throw error listing supported types.
   - Pass through global flags.
