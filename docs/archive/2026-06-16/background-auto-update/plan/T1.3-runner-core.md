# Worker Prompt: T1.3-runner-core

- **Source task**: T1.3 Package source and one-shot update runner core

---

## 1. Mission & Rules

### Mission

Implement the package extraction boundary and one-shot background update runner that refreshes managed skills without updating the CLI package.

### Rules

- Follow the Scope in Section 5 — only modify files listed as Allowed.
- If `pacote` types or runtime imports are unavailable, report the dependency need clearly; do not edit `package.json`.
- Workers are leaf nodes — do not spawn sub-workers.

## 2. Context

### Input Files

- `docs/plans/2026-06-16/background-auto-update/SPEC.md` — Req 3, Req 4, Req 5.
- `docs/plans/2026-06-16/background-auto-update/DESIGN.md` — `skill-update-runner`, `package-source`, invariants.

- `packages/cli/installer.ts` — existing `syncToolkitHome`, `installLinks`, target manifests, overwrite behavior.
- `test/installer.test.js` — installer fixture patterns.

### Background

The runner must update only Apollo Toolkit managed skills/workflows. It must never call `npm install -g` or update the CLI. It should use temp extraction and existing installer semantics so local modified managed skills are overwritten by latest package content.

## 3. Tasks

### `packages/cli/package-source.ts` — create package source boundary

1. Add a new file `packages/cli/package-source.ts`.
2. Define exported types:
   - `PackageSource` with methods `resolveLatest(packageName): Promise<{ version: string; spec: string }>` and `extract(spec, destination): Promise<{ version?: string; sourceRoot: string }>`
   - `PackageSourceOptions` with optional injected functions for tests.
3. Implement a default `createPacotePackageSource()` that dynamically imports `pacote` and calls `manifest` and `extract`.
4. Do not add package dependency here; leave dependency addition to the coordinator/dependency task.

### `packages/cli/auto-update-runner.ts` — create runner

1. Add a new file `packages/cli/auto-update-runner.ts`.
2. Import existing installer functions from `installer.ts`.
3. Import state helpers from `auto-update-state.ts`.
4. Export `runAutoUpdate(options)` with options:
   - `sourceRoot` for current installed package fallback.
   - `toolkitHome`.
   - `packageName`.
   - `currentVersion`.
   - `modes?: InstallMode[]`.
   - `env?: NodeJS.ProcessEnv`.
   - `packageSource?: PackageSource`.
5. Behavior:
   - Acquire `withAutoUpdateLock`.
   - Resolve latest package metadata.
   - If latest version is not newer than current version, write status with enabled state and lastRunAt, then return a no-op success result.
   - Extract latest package into a temp directory.
   - Validate extracted source has expected Apollo Toolkit contents: `package.json` and `skills/`.
   - Call `syncToolkitHome({ sourceRoot: extractedRoot, toolkitHome, version: latestVersion, modes })`.
   - Call `installLinks` with existing target/mode behavior. If modes are not provided, use manifest-aware default strategy or return a clear error requiring modes; choose the simplest behavior consistent with existing target manifests and tests.
   - Write success status including latest version.
   - On failure, write failure status with `lastError` and rethrow or return failure result as appropriate for CLI integration.
   - Always clean temp directory.
6. Explicitly do not import or call `checkForPackageUpdate`.

### `test/cli/auto-update-runner.test.js` — add tests

1. Add a new test file `test/cli/auto-update-runner.test.js`.
2. Use fake `PackageSource` and temp directories.
3. Cover:
   - No-op when latest version is not newer.
   - Managed skill content is overwritten by extracted latest content.
   - Extraction/validation failure preserves previous toolkit home.
   - Status records failure message.
   - Lock prevents concurrent runner mutation.
   - Test asserts no path uses `checkForPackageUpdate` or `npm install -g`.
