## 1. Mission & Rules

### Mission

Implement the config, status, log path, and lock helpers for background auto-update so other modules can persist state under Apollo Toolkit home.

### Rules

- Follow the Scope in Section 5 — only modify files listed as Allowed.
- If a file or function cannot be found, or the implementation is blocked, report it immediately — do not guess or work around it.
- Do not add new dependencies.
- Workers are leaf nodes — do not spawn sub-workers.

### Background

Auto-update config and status must live under Apollo Toolkit home. Missing config means default enabled. Disabled state must persist across future installs. Status must make scheduler or runner failures observable.

## 3. Tasks

### `packages/cli/auto-update-state.ts` — create state module

1. Add a new file `packages/cli/auto-update-state.ts`.
2. Define exported constants:
   - `AUTO_UPDATE_CONFIG_FILENAME = '.apollo-toolkit-auto-update.json'`
   - `AUTO_UPDATE_STATUS_FILENAME = '.apollo-toolkit-auto-update-status.json'`
   - `AUTO_UPDATE_LOG_DIRNAME = 'logs'`
   - `AUTO_UPDATE_LOCK_FILENAME = '.apollo-toolkit-auto-update.lock'`
3. Define exported types:
   - `AutoUpdateConfig` with `enabled: boolean`, `updatedAt: string`.
   - `AutoUpdateStatus` with `enabled: boolean`, `lastRunAt?: string`, `lastSuccessAt?: string`, `lastError?: string`, `lastVersion?: string`, `scheduler?: { registered: boolean; platform: string; message?: string; updatedAt: string }`.
4. Add helpers:
   - `resolveAutoUpdatePaths(toolkitHome: string)` returning absolute paths for config, status, lock, log directory, stdout log, stderr log.
   - `readAutoUpdateConfig(toolkitHome: string): Promise<AutoUpdateConfig>` where missing or malformed config returns `{ enabled: true, updatedAt: <now> }`.
   - `writeAutoUpdateConfig(toolkitHome: string, config: AutoUpdateConfig): Promise<void>` using temp file + rename for atomic write.
   - `readAutoUpdateStatus(toolkitHome: string): Promise<AutoUpdateStatus | null>` where missing or malformed status returns null.
   - `writeAutoUpdateStatus(toolkitHome: string, status: AutoUpdateStatus): Promise<void>` using temp file + rename.
   - `withAutoUpdateLock<T>(toolkitHome: string, fn: () => Promise<T>): Promise<T>` that creates the lock file with exclusive open and removes it in `finally`; if lock exists, throw an Error with a clear message.
5. Use only Node standard library APIs. Preserve the repo's ESM TypeScript style.

### `test/cli/auto-update-state.test.js` — add tests

1. Add a new test file `test/cli/auto-update-state.test.js`.
2. Import from built output path consistent with existing tests after build, or match local test conventions for TypeScript packages.
3. Cover:
   - Missing config defaults to enabled.
   - Disabled config reads back disabled.
   - Malformed config falls back to enabled without throwing.
   - Status writes and reads back.
   - Lock prevents a second concurrent lock acquisition.
   - Lock is removed after the callback resolves or rejects.

### Output

When done, report back to the coordinator:
- **Files modified**: list files.
- **Change summary**: summarize state helpers and tests.
- **Test results**: pass/fail for verification commands.
- **Risks or concerns**: or "None".
